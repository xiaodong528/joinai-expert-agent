# 工程建设审核智能体场景业务流程文档

> 面向读者：业务负责人、方案经理、售前/交付团队
>
> 文档定位：描述工程建设审核智能体的目标态业务闭环、参与角色、输入输出和异常处理方式
>
> 说明：本文按目标态能力编写，用于统一业务理解与方案表达，不展开脚本、技能目录或底层实现细节

## 1. 文档目的

工程建设审核智能体用于把原本依赖人工逐项核对的预算审核、结算审核流程，转化为一条可复制、可追溯、可规模化运行的自动化审核业务链路。它面向的不是“单张表算错了怎么修”，而是“从用户提交资料到交付审核结论”的完整业务闭环。

这份文档重点回答六个问题：

1. 这个场景解决什么问题。
2. 谁参与这条业务流程。
3. 发起审核前需要准备什么资料。
4. 系统在中间做了哪些业务动作。
5. 最终会交付什么结果。
6. 遇到资料异常、规则歧义或高风险问题时如何处理。

## 2. 场景背景与业务目标

在建设工程造价审核场景中，预算书和结算书通常包含多张相互引用的工作表，审核依据则散落在规则文档、费率说明和项目约束中。传统人工审核存在三个典型问题：

- 审核耗时长。一个项目往往需要 30 分钟以上，复杂项目更长。
- 漏审和误审风险高。跨表引用、费率分支、零值规则等场景容易被忽略。
- 结果不稳定。不同审核人员对规则理解、检查深度和报告表达方式不一致。

工程建设审核智能体的业务目标是：

- 将审核流程标准化，降低对个人经验的依赖。
- 将规则理解、表格拆解、差异识别和报告交付串成一条完整链路。
- 用并行处理缩短审核周期，让单项目审核从“人工逐项核对”变成“系统批量复核”。
- 形成可回溯的审核证据，让业务方能够追踪每条问题的依据、位置和处理结果。

## 3. 适用对象与业务边界

### 3.1 适用对象

本文档描述的业务流程优先适用于以下场景：

- 家客电信建设工程预算审核
- 家客电信建设工程结算审核
- 具有规则文档和多 Sheet 工程表格的同类造价审核场景

在后续扩展中，该流程也可迁移到房建、市政或其他具备“规则文档 + 工程表格”双输入特征的审核业务。

### 3.2 适用边界

系统重点处理以下问题：

- 规则理解与审核依据生成
- 多 Sheet 表格结构化与交叉校验
- 计算类错误、费率类错误、条件分支错误、跨表引用错误识别
- 审核报告生成与修正结果交付

以下内容不作为本文主流程重点：

- 纯样式类、纯排版类问题
- 缺乏业务依据的主观判断项
- 与规则文档完全无关的额外经营决策

### 3.3 业务原则

- 正式输入以用户提交的规则文档和待审工程表格为准，不依赖系统内置固定规则库替代用户资料。
- 原始表格不直接被覆盖修改，系统输出的是审核结论和基于副本生成的修正结果。
- 对能够确定性判定的问题，系统给出明确差异；对存在规则歧义或业务前提不足的问题，系统转交人工复核。

## 4. 参与角色与职责

| 角色 | 职责定位 | 业务职责 |
| --- | --- | --- |
| 审核发起人 | 外部业务角色 | 提交规则文档、工程表格、审核类型和项目背景，接收最终交付物 |
| 编排者 | 流程总控角色 | 建立审核会话、安排各处理环节顺序、控制阶段门禁、汇总最终交付 |
| 执行者 | 处理角色 | 执行资料校验、规则解析、表格拆解、按 Sheet 审核、报告组装 |
| 审查者 | 质量角色 | 复核中间产物与最终结论，检查遗漏、异常和高风险问题 |
| 监控者 | 运行保障角色 | 监控流程状态、识别超时/失败/中断，并推动恢复或升级处理 |

从业务视角看，这五类角色共同组成“提交资料 -> 自动审核 -> 质量复核 -> 结果交付”的闭环。其中审核发起人负责提供上下文，其余四类角色由系统内部协同完成。

## 5. 前置条件与输入资料

审核任务开始前，需准备以下资料：

| 输入项 | 说明 | 必填 |
| --- | --- | --- |
| `rule_document.docx` | 审核规则文档，包含费率、计算口径、分支条件、不计取规则等 | 是 |
| `spreadsheet.xlsx/.xls` | 待审核的工程预算书或结算书，通常为多 Sheet 工作簿 | 是 |
| 审核类型 | 预算审核或结算审核 | 是 |
| 项目名称/标识 | 用于会话建档、报告展示和归档追踪 | 是 |
| 特殊范围说明 | 例如只审核指定表、指定章节或指定费用类别 | 否 |

资料进入流程前需满足以下前置条件：

- 规则文档可读取，且能识别出基本结构。
- 工程表格可读取，且工作表结构完整。
- 审核类型明确，避免预算/结算口径混用。
- 若存在仅审部分表格或附加业务约束，需要在任务发起时说明。

## 6. 端到端业务流程

### 6.1 流程总览

```mermaid
flowchart LR
    A["发起审核任务"] --> B["提交规则文档与工程表格"]
    B --> C["会话建档与资料校验"]
    C --> D["生成审核依据"]
    D --> E["拆解工作簿并分发表格任务"]
    E --> F["按 Sheet 并行审核"]
    F --> G["汇总问题并生成报告"]
    G --> H["交付审核报告与修正结果"]
    H --> I["人工确认与归档"]
```

### 6.2 分步说明

| 步骤 | 业务动作 | 系统响应 | 阶段输出 |
| --- | --- | --- | --- |
| 1. 发起审核任务 | 审核发起人提交项目背景、审核类型和范围说明 | 系统创建本次审核会话，记录任务标识与处理范围 | 审核会话、任务编号 |
| 2. 提交审核资料 | 审核发起人上传规则文档和工程表格 | 系统接收正式输入并进入校验环节 | 原始输入资料归档记录 |
| 3. 会话建档与资料校验 | 系统检查文件可读性、格式和基本结构 | 若资料齐备则进入下一步；若缺失或损坏则中止并反馈 | 会话配置、资料校验结果 |
| 4. 生成审核依据 | 系统阅读规则文档，提炼审核口径、费率规则、条件分支和不计取规则 | 形成可追溯的审核依据，用于后续逐表复核 | 审核依据集 |
| 5. 工作簿拆解与任务分发 | 系统识别工作簿中的工作表、公式关系和关键字段 | 将复杂工作簿拆成可并行处理的审核单元 | 表格结构化结果、待处理任务清单 |
| 6. 按 Sheet 并行审核 | 执行者并行审核各工作表，识别差异、定位依据、判断严重度 | 审查者同步复核关键问题，控制漏审与误判风险 | 分表问题清单 |
| 7. 汇总问题并生成报告 | 系统合并所有审核发现，整理问题分类、原因说明和修正建议 | 生成统一的业务交付材料 | 审核报告、结构化问题汇总、修正结果副本 |
| 8. 交付与归档 | 审核发起人接收审核结果，进行人工确认 | 系统保留审核过程记录，支撑追溯和复盘 | 交付完成记录、归档记录 |

### 6.3 关键业务说明

#### 步骤 1：发起审核任务

这一环节的核心不是上传文件，而是明确“本次审核要解决什么问题”。系统需要知道是预算审核还是结算审核、是否仅覆盖某些表、是否存在特殊业务口径。范围界定越清晰，后续审核结果越稳定。

#### 步骤 2：提交审核资料

规则文档与工程表格构成正式输入契约。业务上必须保证规则依据和待审数据来自同一审核语境，避免“旧规则审新表”或“预算口径审结算表”的情况。

#### 步骤 3：会话建档与资料校验

这是流程的第一道门禁。系统会在这一阶段确认资料是否可读、工作簿是否完整、会话是否具备继续处理的条件。如果基础资料不成立，流程不会进入“自动审核”，而是直接返回资料问题，避免产生看似完整但依据失真的结果。

#### 步骤 4：生成审核依据

系统将规则文档转化为可执行的审核依据，包括但不限于：

- 费率口径
- 公式关系
- 条件分支
- 不计取项
- 需要人工确认的特殊情形

这一环节的业务意义在于，把“文字规则”转换成“可批量复核的审核标准”，同时保留依据来源，保证后续每一条问题都能回溯到规则出处。

#### 步骤 5：工作簿拆解与任务分发

工程表格往往由多张工作表组成，且相互引用。系统会先识别结构，再把审核工作拆成多个可独立处理的单元，以便并行处理。业务上，这意味着系统能够同时审查多个表，而不是像人工那样顺序核对。

#### 步骤 6：按 Sheet 并行审核

这是价值最集中的环节。系统会围绕每张表中的费用项、合计项、费率项和跨表关联项做逐项复核，输出的问题会包含：

- 问题位置
- 实际值与期望值
- 触发该问题的审核依据
- 严重程度
- 是否可自动修正或需人工确认

对于确定性问题，系统给出明确差异；对于规则歧义、数据缺口或业务上下文不足的问题，系统会标记为“需人工复核”，而不是强行下结论。

#### 步骤 7：汇总问题并生成报告

系统将各工作表的审核结果汇总成统一交付件。对业务方而言，这一步的目标不是“把所有技术细节都展开”，而是输出一份可以直接阅读、转发和归档的审核成果，通常包括：

- 审核报告
- 结构化问题清单
- 修正结果副本

#### 步骤 8：交付与归档

自动审核的最终落点仍然是业务闭环。审核发起人会基于报告进行确认、复核或转办。系统则保留过程记录，用于后续复盘、争议追踪和规则优化。

## 7. 关键决策点与异常分支

业务流程中存在四类关键分支，需要明确处理策略：

### 7.1 资料不完整或不可读

- 触发条件：规则文档损坏、工程表格无法读取、关键工作表缺失、审核类型未说明
- 处理方式：流程在资料校验阶段中止，返回明确补件要求
- 业务原则：不以不完整资料进入自动审核

### 7.2 规则存在歧义或缺乏上下文

- 触发条件：规则表述冲突、费率前提不明确、文档无法唯一映射到目标数据项
- 处理方式：系统保留原始依据并标记为“需人工复核”
- 业务原则：不在依据不足时强行给出确定性结论

### 7.3 并行审核中发现高风险问题

- 触发条件：关键费用项异常、跨表关联大面积不一致、疑似高估冒算
- 处理方式：审查者提升问题等级，必要时阻断自动交付，转人工确认
- 业务原则：高风险问题优先保证准确性，而不是追求全自动结束

### 7.4 部分工作表处理失败

- 触发条件：单表结构异常、公式读取失败、关键数据缺失
- 处理方式：系统阻断最终报告发布，提示失败范围与原因
- 业务原则：最终交付必须建立在完整处理结果之上，不能隐含漏审

## 8. 输出结果与业务价值

### 8.1 标准交付物

在目标态业务闭环中，系统默认输出以下结果：

| 交付物 | 面向对象 | 用途 |
| --- | --- | --- |
| 审核报告 | 业务负责人、审核人员、交付团队 | 统一查看问题分类、问题依据、风险等级和处理建议 |
| 结构化问题清单 | 系统内留档、后续分析 | 支撑追溯、统计、规则优化和质量复盘 |
| 修正结果副本 | 审核执行团队 | 用于对照问题项与修正建议，不覆盖原始文件 |
| 审核归档记录 | 管理与复盘角色 | 留存输入资料、审核过程和最终交付的关联关系 |

### 8.2 业务价值

工程建设审核智能体在业务上的核心价值体现在四个方面：

- 效率提升：将串行、人工主导的审核流程转为可并行的自动化处理链路。
- 质量稳定：统一规则理解和问题表达方式，降低漏审和口径不一致风险。
- 风险可控：对高风险问题保留人工复核门禁，避免“自动化放大错误”。
- 过程可追溯：每条问题都有依据、有位置、有输出，便于后续复核和申诉处理。

## 9. 建议的落地使用方式

面向业务团队，建议按以下方式推动落地：

1. 先选取规则清晰、表格结构稳定的预算/结算样例做试点。
2. 将审核报告作为正式业务交付件，统一问题口径和输出模板。
3. 对“需人工复核”问题建立回写机制，逐步完善规则文档质量。
4. 将高频问题和人工判定结果沉淀回规则体系，持续提升自动审核覆盖率。

## 10. 结语

工程建设审核智能体的价值不在于替代所有人工判断，而在于把最耗时、最重复、最容易遗漏的审核动作系统化。对业务团队而言，它提供的是一条可规模复制的审核作业链；对方案团队而言，它提供的是一套可说明、可交付、可持续优化的业务流程框架。
