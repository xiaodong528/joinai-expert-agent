## 1. 执行摘要

JAS（JoinAI Swarm Factory）视频生成智能体是一套端到端全自动化的 AI 视频制作系统。它将传统视频制作中的编剧、分镜、画面生成、配音、配乐、剪辑、字幕、质检等环节，整合为 11 个自动化阶段（Stage 0–10），由多个 AI 智能体协同并行执行，实现从"一句话创意"到"成品视频"的全自动交付。

系统支持 **5 种以上视频类型**——短剧、宣传片/广告、教程/解说、创意短视频、Vlog/纪录——并根据视频类型**自动裁剪管线**，跳过不需要的阶段，既节省成本又缩短周期。

## 2. 系统架构总览

### 2.1 三层架构

JAS 视频生成智能体采用**三层分离架构**，将调度、逻辑、执行各司其职：

```
┌─────────────────────────────────────────────────────┐
│  GasTown Mayor          并行编排层                    │
│  ── 统筹调度、Wave 并行分发、质量门控                    │
├─────────────────────────────────────────────────────┤
│  OpenCode Agent: video-generation-orchestrator    编排逻辑层          │
│  ── 管线裁剪、子智能体派发、串行收尾                     │
├─────────────────────────────────────────────────────┤
│  11 个 Stage Skill                执行层              │
│  ── 每个 Skill 含独立脚本 + API 客户端，完全自包含       │
└─────────────────────────────────────────────────────┘
```

- **编排层（Mayor）**：不执行具体生成任务，专注于调度决策、Wave 分发和质量检查
- **逻辑层（Agent）**：根据视频类型裁剪管线，将任务分配给子智能体（Polecat）并行执行
- **执行层（Skill）**：每个 Stage 封装为独立 Skill，含 Python 脚本和 API 客户端，互不依赖

![三层架构：Mayor → Agent → Skill](illustrations/jas-video-agent-report/02-framework-three-layer-arch.png)

### 2.2 11 阶段管线

| 阶段 | 名称       | 属性         | 执行者          | 核心产出                                 |
| ---- | ---------- | ------------ | --------------- | ---------------------------------------- |
| 0    | 创意策划   | 必选         | Mayor（交互式） | story.yaml（剧本 + 角色 DNA + 风格锁定） |
| 1    | 分镜脚本   | 必选         | Polecat         | storyboard.yaml（N 个分镜记录）          |
| 2    | 角色锚定   | 可选         | Polecat         | 角色参考图（确保跨场景外貌一致）         |
| 3    | 关键帧生成 | 必选         | Polecat ×N     | 每场景首帧 + 尾帧图像                    |
| 4    | 图生视频   | 必选         | Polecat ×N     | 每场景 12 秒视频片段                     |
| 5    | TTS 语音   | 可选         | Polecat         | 台词/旁白 MP3 + 字幕时间戳               |
| 6    | BGM 配乐   | 可选（推荐） | Polecat         | 纯音乐背景音轨                           |
| 7    | 视频拼接   | 必选         | Mayor           | 多片段合并 + 音轨混合                    |
| 8    | 对口型     | 可选         | Mayor           | 角色嘴型与台词同步                       |

| 10   | 质量审查   | 必选         | Mayor           | 自动化 QA 报告（JSON）                   |

### 2.3 Mayor-Polecat 分工模型

系统采用 **Mayor + Polecat** 的主从智能体架构：

- **Mayor（主智能体）**：负责交互式策划（Stage 0）和确定性串行操作（Stage 7-10），以及全局调度和质量门控
- **Polecat（子智能体）**：负责创造性生成和耗时的 AI 模型调用（Stage 1-6），每个 Polecat 只承担一个具体任务，完成后向 Mayor 报告

这种设计确保 Mayor 保持轻量决策能力，不被大量生成计算占用；Polecat 完全无状态，通过共享文件系统传递数据，支持弹性扩展。

## 3. 多智能体并行架构 — Wave 调度模型

Wave 调度模型是 JAS 视频生成智能体的核心架构创新。它将 11 个阶段组织为**三个并行波浪（Wave）+ 一个串行收尾阶段**，最大化利用 AI 模型的并行处理能力。

### 3.1 Wave 并行化流程

```
                    ┌──────────────┐
                    │   Stage 0    │  Mayor 与用户交互式策划
                    │   创意策划    │  输出: story.yaml（冻结）
                    └──────┬───────┘
                           ▼
              ┌────────────────────────┐
    Wave 1    │  P1: Stage 1 分镜脚本   │  2 个 Polecat 并行
              │  P2: Stage 2 角色锚定   │  （角色锚定仅短剧启用）
              └────────────┬───────────┘
                    质量门控 ▼ 分镜验证（13 项结构化检查）
              ┌────────────────────────┐
    Wave 2    │  Stage 3 关键帧生成     │  N 个 Polecat
              │  首帧锚定链（批内串行）  │  （可分批并行）
              └────────────┬───────────┘
                    质量门控 ▼ 角色一致性抽查
    ┌──────────────────────────────────────────────┐
    │  P1..PN: Stage 4 图生视频（每场景 1 个 Polecat）│
    │  P(N+1)..P(2N): Stage 5 TTS（每场景 1 个）     │  2N+1 个 Polecat
    │  P(2N+1): Stage 6 BGM（全局 1 个）             │  完全并行
    Wave 3                                           │
    └──────────────────────┬───────────────────────┘
                    质量门控 ▼ 时长和完整性检查
              ┌────────────────────────┐
    串行阶段   │  Stage 7 → 8 → 9 → 10 │  Mayor 依次执行
              │  拼接→对口型→字幕→QA    │
              └────────────────────────┘
```

**设计哲学**：耗时的 AI 生成任务（图像、视频、语音、音乐）全部并行化，确定性操作（FFmpeg 拼接、字幕烧录）串行执行。Wave 之间设置同步屏障和质量门控，确保上游输出质量后再启动下游。

![Wave 并行调度模型流程图](illustrations/jas-video-agent-report/03-flowchart-wave-scheduling.png)

### 3.2 并行度扩展能力

随着场景数增加，系统的并行度**线性扩展**：

| 场景数 N | Wave 2 Polecat | Wave 3 Polecat | 总 Polecat 数 | 峰值并行度   |
| -------- | -------------- | -------------- | ------------- | ------------ |
| 6        | 6              | 13             | 21            | **13** |
| 10       | 10             | 21             | 33            | **21** |
| 20       | 20             | 41             | 63            | **41** |
| 30       | 30             | 61             | 93            | **61** |

> 峰值并行度出现在 Wave 3（图生视频 + TTS + BGM 全部同时启动）。10 场景的标准短剧，最高有 **21 个 AI 智能体同时工作**，类似一条有 21 个工位的智能流水线。

![并行度线性扩展能力](illustrations/jas-video-agent-report/04-infographic-scalability.png)

### 3.3 首帧锚定链 — 视觉连贯性保障

为确保相邻场景之间的视觉连贯性，系统采用**首帧锚定链**机制：

- **场景 1 首帧**：由文本提示词直接生成（无依赖）
- **场景 2+ 首帧**：从前一场景的尾帧通过"图生图"变换派生，保留 60-70% 的视觉元素

这类似于电影分镜师在每个镜头切换点确保视觉连贯：角色外貌、光线色调、环境氛围从上一个镜头自然过渡到下一个镜头。

**实测结论**：在 architect-encounter 项目中，5 个场景的锚定链传递后，风格漂移极小，角色一致性保持优秀。在 cmcc-industrial 项目中，10 个场景的锚定链传递同样保持了 3D 科技风格的一致性，验证了该机制在更大规模下的稳定性。

### 3.4 质量门控机制

管线设置 **4 个质量检查点**，确保每个 Wave 的输出质量：

| 检查时机      | 检查内容                                  | 不通过处理         |
| ------------- | ----------------------------------------- | ------------------ |
| Wave 1 完成后 | 分镜脚本结构化验证（4 类 13 项检查）      | 自动修正或重新生成 |
| Wave 2 完成后 | 抽查前 2-3 场景首帧的角色一致性           | 调整参数后重跑     |
| Wave 3 完成后 | 视频时长、音频完整性检查                  | 重试失败场景       |
| Stage 10      | 自动化 QA（分辨率、时长、编码、黑帧检测） | 生成修复建议       |

### 3.5 容错与弹性

- **幂等可重试**：所有输出按场景编号命名，同一场景可安全重跑，不影响其他场景
- **备用模型切换**：每个关键环节配备次选方案（Seedance → Wan 2.2、TTS → CosyVoice、Music → Mureka），API 异常时自动切换
- **部分失败隔离**：单个 Polecat 失败不影响其他并行任务，Mayor 仅重试失败的 Bead

## 4. 视频类型与管线裁剪

### 4.1 支持的视频类型

系统预置 **5 种视频类型**，每种类型自动启用/跳过对应的 Stage，实现管线裁剪：

| 视频类型              | 必选阶段           | 可选阶段（建议启用） | 可跳过阶段 | 典型时长      |
| --------------------- | ------------------ | -------------------- | ---------- | ------------- |
| **短剧**        | 0,1,2,3,4,5,6,7,10 | 8,9                  | —         | 1–10 分钟    |
| **宣传片/广告** | 0,1,3,4,7,10       | 5,6,9                | 2,8        | 15 秒–3 分钟 |
| **教程/解说**   | 0,1,3,4,5,7,10     | 9                    | 2,6,8      | 3–15 分钟    |
| **创意短视频**  | 0,1,3,4,7,10       | 6                    | 2,5,8,9    | 15–60 秒     |
| **Vlog/纪录**   | 0,1,3,4,5,7,10     | 6,9                  | 2,8        | 3–10 分钟    |

![视频类型管线裁剪矩阵](illustrations/jas-video-agent-report/05-comparison-pipeline-trim.png)

### 4.2 Wave 裁剪映射

管线裁剪直接影响每个 Wave 的执行内容和资源消耗：

| 视频类型   | Wave 1          | Wave 2 | Wave 3           | Mayor 串行             |
| ---------- | --------------- | ------ | ---------------- | ---------------------- |
| 短剧       | 分镜 + 角色锚定 | 关键帧 | 视频 + TTS + BGM | 拼接→对口型→字幕→QA |
| 宣传片     | 仅分镜          | 关键帧 | 视频 + BGM       | 拼接→字幕→QA         |
| 创意短视频 | 仅分镜          | 关键帧 | 视频 + BGM       | 拼接→QA               |

**关键差异**：

- **角色锚定**（Stage 2）仅短剧启用——宣传片和创意短视频通常无固定角色
- **TTS 语音**（Stage 5）仅有台词/旁白的视频类型启用
- **对口型**（Stage 8）仅短剧且有面部特写场景时启用，是成本最高的可选环节

### 4.3 自适应扩展能力

- **场景数弹性**：支持 3–30 个场景，系统自动调整并行度（Wave 3 Polecat 数量 = 2N+1）
- **时长弹性**：单场景 ≤10 秒，总时长从 15 秒短片到 10 分钟中长视频均可覆盖
- **新类型扩展**：只需在 Stage 0 定义新的视频类型预设（启用/跳过的 Stage 列表），Mayor 自动适配调度逻辑，无需修改底层代码

## 6. 成本与效率分析

### 6.1 单片成本拆解

| 成本项                        | 金额 (¥)               | 占比 | 说明                        |

> **成本优化空间**：Lip-Sync 是最大成本项（占 40%+），仅对有人物面部特写的场景启用。若不启用 Lip-Sync 和角色锚定，仅必选阶段的最低成本可降至 **¥3–7/片**。

### 6.3 成本优化策略

3. **批量生产调度**：高并发时 Seedance（10 并发）+ 可灵（5 QPS）溢出队列混合调度

### 7.2 选型原则

| 原则                   | 说明                                       |
| ---------------------- | ------------------------------------------ |
| **API/CLI Only** | 所有模型必须有可编程接口，确保全自动化     |
| **中国模型优先** | 国内合规 + 中文效果最优 + 延迟更低         |
| **成本可控**     | 单片总成本控制在 ¥20 以内                 |
| **容灾备份**     | 每个关键环节配备次选方案，API 异常时可切换 |

### 7.3 供应商依赖

系统依赖 **4 个云服务平台**，共 **6 组 API 密钥**：

| 平台                 | 覆盖环节                            | 角色             |
| -------------------- | ----------------------------------- | ---------------- |
| **火山引擎**   | 图像 + 视频 + TTS + ASR（4 个环节） | 核心依赖         |
| **MiniMax**    | BGM 音乐生成                        | 单一环节         |
| **快手可灵**   | Lip-Sync 对口型                     | 单一环节（可选） |
| **阿里云百炼** | 备用视频生成（Wan 2.2）             | 容灾备份         |

> 火山引擎覆盖最多环节，是系统的核心供应商。其余平台各覆盖单一环节，供应商切换成本低。
